# Слайд 1. Введение: что сделано

## Что на слайде
Краткое изложение продукта: система realtime-инференса для Dota 2, которая на основе Game State Integration (GSI) предсказывает вероятность гибели героя и строит тепловую карту риска.

## Что говорить
Здравствуйте. Меня зовут [Ваше имя]. Я представляю проект — систему realtime-инференса событий в Dota 2. Проект принимает данные от локального Game State Integration, преобразует их в признаки, передаёт в очередь, затем модель предсказывает вероятность того, что герой умрёт в ближайшие N секунд, и публикует как индивидуальные предсказания, так и тепловую карту риска на игровом поле. Основные компоненты: GSI-обработчик, очередь на Redis Streams, сервис инференса на FastAPI с воркером и набор моделей (CatBoost и логистические пайплайны). В ходе защиты я покажу архитектуру, мотивацию, ключевые инженерные решения и демонстрацию сценария использования.

## Ключевые тезисы
- realtime-инференс на основе GSI
- CatBoost + логрег как бэкенды моделей
- Redis Streams для надёжной передачи сообщений
- 32x32 heatmap с публикацией в Redis

## Технические детали / акценты
- Источники: [readme.md](readme.md#L1-L200) — итоговые метрики моделей
- Код инференса: [services/inference/scr/services/model.py](services/inference/scr/services/model.py#L1-L200)

## Что на слайде

## Ключевые тезисы
- Цель: предсказание смерти в ближайшие N секунд
- Метрики: ROC-AUC / PR-AUC / logloss (см. `readme.md`)

# Слайд 4. Функциональность и сценарии использования

## Что на слайде
Список сценариев: GSI → preprocessing → enqueue → inference → heatmap + output stream; примеры использования (HUD, аналитика).

## Что говорить
Сценарии: (1) GSI клиент отправляет состояния через HTTP, (2) `gsi_processor` вычисляет признаки, дополняет информацией о противниках/башнях/предметах, создаёт `InferenceRecord` и ставит его в Redis Stream, (3) сервис инференса читает сообщения группой потребителей, строит батчи, вызывает модель (CatBoost или логрег-пайплайн), (4) результаты отправляются в выходной Stream, а для тепловой карты генерируется grid-предсказание для 32×32 клеток и сохраняется/публикуется в Redis с TTL.

## Ключевые тезисы
- Интеграция через Redis Streams
- Batch-обработка и fallback на default-модель
- Heatmap rebuild per tick, TTL + pub/sub уведомление

## Технические детали / акценты
- GSI процессинг: [services/gsi_processor/scr/services/process.py](services/gsi_processor/scr/services/process.py#L1-L200)

# Слайд 5. Архитектура системы

## Что на слайде
Архитектурная диаграмма: GSI → GSI Processor → Redis Streams (in) → Inference Worker (FastAPI) → Redis Streams (out) + Heatmap key → Frontend/BFF.

## Что говорить
Ключевые компоненты: GSI-обработчик, Redis (Streams, key-value + pub/sub), Inference Service (FastAPI), модельный реестр и предикторы. Такая архитектура даёт слабую связанность между источником данных и моделью, обеспечивает устойчивость (механика потребительских групп Redis Streams) и лёгкую горизонтальную масштабируемость воркеров. Frontend/BFF может подписываться на ключи heatmap через Redis pub/sub для быстрого обновления визуализации.

## Ключевые тезисы
- Модульность: ingestion, preprocessing, inference, delivery
- Надёжность: Redis Streams + consumer groups
- Масштабирование: можно добавлять воркеры, разделить модели по директориям артефактов

# Слайд 6. Технологический стек

## Что на слайде
Краткий список технологий: Python, FastAPI, Pydantic, Redis (Streams), CatBoost, scikit-learn/joblib, pandas, Docker.

- Redis Streams как backbone событий

# Слайд 7. Машинное обучение: модели и pipeline

## Что на слайде
Описание моделей (CatBoost vs логрег), почему выбран CatBoost, как устроен pipeline от GSI до инференса, важные метрики из README.

## Что говорить
Для задачи бинарного события в малом горизонте времени CatBoost — хороший выбор: естественная работа с категориальными признаками (герои), хорошая производительность и относительная простота деплоя (`.cbm` модель загружается напрямую). Пайплайн подготовки признаков реализован в `gsi_processor` (нормализация, упаковка предметов/способностей, расстояния, cell_id), затем объекты сериализуются и отправляются в очередь. В README представлены метрики для нескольких горизонтов: ROC-AUC для CatBoost достигает ~0.84–0.80 в зависимости от горизонта, что показывает полезность модели в задачах локального прогноза.

## Ключевые тезисы
- CatBoost: robust to categorical features (hero_id, enemy names)
- Логрег-пайплайны для сравнения / легкой интерпретации
- Метрики: ROC-AUC ~0.8+ для дальних горизонтов (см. README)

# Слайд 8. Ключевые инженерные решения и компромиссы

## Что на слайде
Список сильных решений и принятых компромиссов: Redis Streams, model registry, grid-inference для heatmap, использование pandas в runtime и связанные с этим trade-offs.

## Что говорить
Сильные решения: использование Redis Streams обеспечивает устойчивость очереди и упрощает горизонтальное масштабирование; ModelRegistry позволяет хранить несколько моделей по времени и типу; grid-inference даёт удобную heatmap-агрегацию для визуализации. Компромиссы: вычисление grid-прогнозов для каждой клетки увеличивает нагрузку — проект компенсирует это ограничением размера сетки (32×32) и выбором интервала/радиуса среза; внутри предикторов используется pandas, что удобно, но даёт накладные расходы на latency; выбор файловой загрузки моделей (cbm/joblib/pickle) упрощает деплой, но требует согласованности версий библиотек.

## Ключевые тезисы
- Плюсы: надёжность, модульность, готовность к масштабированию
- Минусы: latency при полной heatmap-инференции, зависимость от pandas/CatBoost runtime

## Технические детали / акценты
- Возможный узкий горлышко: предсказание для full_map — строится полная сетка, см. [worker.py](services/inference/scr/services/worker.py#L1-L200)


# Слайд 9. Ограничения и направления развития

## Что на слайде
Короткий перечень ограничений и roadmap: оптимизация latency, model serving (ONNX / C API), масштабирование, мониторинг и A/B тестирование.

## Что говорить
Ограничения: текущая реализация предполагает Python runtime с pandas и CatBoost — это влияет на latency и требования к среде. Heatmap по умолчанию строится заново каждый тик и хранится в Redis с коротким TTL, что может быть неэффективно при большом количестве токенов. Дальнейшие шаги: (1) экспорт моделей в ONNX или нативный C-рантайм для уменьшения latency, (2) шардирование очередей и независимые воркеры на топики с разными SLA, (3) наблюдаемость: метрики производительности, latency, ошибок, (4) улучшение данных — больше матчей, слои feature-engineering, онлайн-обучение/перетренировка.

## Ключевые тезисы
- Performance: ONNX/LibCatBoost C API
- Scalability: sharding, autoscaling воркеров
- Observability: Prometheus/Tracing

## Технические детали / акценты
- Предложение: заменить pandas в predict_proba на быстрые numpy-матрицы или Numba-преобразования для критичной нагрузки
